## 1.1 后端框架：FastAPI vs Flask vs Django

| 维度 | FastAPI | Flask | Django |
|------|---------|-------|--------|
| 异步支持 | ✅ 原生asyncio | ⚠️ 需额外库 | ⚠️ 有限支持 |
| 自动API文档 | ✅ 内置Swagger/Redoc | ❌ 需手动写 | ⚠️ DRF可生成 |
| 数据验证 | ✅ Pydantic强制 | ❌ 需手动 | ✅ DRF Serializer |
| 学习曲线 | 中 | 低 | 高 |
| 性能 | 高（异步） | 中 | 中 |
| 社区/生态 | 中（较新） | 大 | 极大 |
| AI集成友好度 | ✅ 异步流式响应天然支持 | ⚠️ 需封装 | ⚠️ 略重 |

**结论：选FastAPI ✅**

**理由**：
1. 原生异步，完美支持Claude API的流式响应（Streaming）
2. 自动生成Swagger文档，开发效率高
3. Pydantic数据验证，减少Bug
4. 性能高，单服可支撑1000+并发

**代价**：社区比Django小，遇到坑需要自己查（但用AI辅助可解决）

## 1.2 前端框架：React vs Vue vs Angular

| 维度 | React + Vite | Vue 3 + Vite | Angular |
|------|--------------|--------------|---------|
| 学习曲线 | 中（JSX） | 低（模板语法） | 高（TypeScript强制） |
| 生态 | 极大 | 大 | 中 |
| 移动端适配 | ✅ React Native | ✅ Vue Native | ❌ 无 |
| AI组件库 | ✅ 丰富 | ✅ 丰富 | ⚠️ 一般 |
| 打包速度 | 快（Vite） | 快（Vite） | 慢 |
| 招聘市场需求 | 最高 | 高 | 中 |

**结论：选React + Vite + Ant Design ✅**

**理由**：
1. 生态最大，遇到问题Google/AI都能找到答案
2. Ant Design组件库成熟，快速搭建后台界面
3. Vite打包速度快，开发体验好
4. 未来可扩展React Native做移动端（高考志愿场景移动端很重要）

**代价**：JSX语法需要适应（但用AI写JSX不难）

## 1.3 数据库：PostgreSQL vs MySQL vs MongoDB

| 维度 | PostgreSQL | MySQL 8.0 | MongoDB |
|------|-------------|-----------|---------|
| JSON支持 | ✅ 原生JSONB | ⚠️ 有限 | ✅ 原生 |
| 全文搜索 | ✅ 内置 | ⚠️ 需插件 | ❌ 无 |
| 向量搜索 | ✅ pgvector扩展 | ❌ 无 | ⚠️ 需Atlas |
| 事务完整性 | ✅ ACID | ✅ ACID | ⚠️ 最终一致 |
| 云服务支持 | ✅ 阿里云RDS | ✅ 阿里云RDS | ✅ 阿里云MongoDB |
| 成本 | 中 | 低 | 高 |
| 运维复杂度 | 中 | 低 | 中 |

**结论：选PostgreSQL ✅（开发阶段用SQLite，上线换PostgreSQL）**

**理由**：
1. 未来可能加向量搜索（pgvector），PostgreSQL原生支持
2. JSONB类型适合存储灵活的"用户画像"数据
3. 全文搜索内置，未来做"院校搜索"可直接用
4. 开发阶段用SQLite（零配置），上线切PostgreSQL（改连接字符串即可）

**代价**：PostgreSQL比MySQL略难运维（但用阿里云RDS可托管）

## 2.2 users 表（用户账户）

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  phone VARCHAR(20) UNIQUE NOT NULL,  -- 手机号（登录名）
  password_hash VARCHAR(128) NOT NULL,  -- bcrypt加密后的密码
  nickname VARCHAR(50),                 -- 昵称（可选）
  avatar_url VARCHAR(200),              -- 头像URL（可选）
  role VARCHAR(20) DEFAULT 'student',  -- 角色: student/parent/teacher/admin
  created_at TIMESTAMPTZ DEFAULT NOW(),
  last_login TIMESTAMPTZ,
  is_active BOOLEAN DEFAULT TRUE,
  CONSTRAINT chk_role CHECK (role IN ('student','parent','teacher','admin'))
);

-- 索引
CREATE INDEX idx_users_phone ON users(phone);
CREATE INDEX idx_users_created_at ON users(created_at);
```

**字段说明**：
- `phone`：唯一，用于登录（MVP阶段无短信验证，密码登录）
- `password_hash`：bcrypt加密（cost=12），明文密码永远不存
- `role`：学生/家长/老师/管理员（MVP只做student）
- `is_active`：软删除标志（不真正删用户数据）

## 2.3 exams 表（考试记录）

```sql
CREATE TABLE exams (
  id SERIAL PRIMARY KEY,
  user_id INTEGER REFERENCES users(id) ON DELETE CASCADE,
  exam_name VARCHAR(100) NOT NULL,  -- 考试名称（如"高二上学期期中考试"）
  exam_date DATE NOT NULL,            -- 考试日期
  total_score INTEGER,                -- 总分（原始分）
  total_full INTEGER,                 -- 满分（如750/660/900）保留原始总分
  standard_score INTEGER,             -- 折算成750分制的分数
  subject_scores JSONB,              -- 各科分数（{"数学":120,"英语":110}）
  analysis_status VARCHAR(20) DEFAULT 'pending',  -- pending/done/failed
  created_at TIMESTAMPTZ DEFAULT NOW(),
  CONSTRAINT fk_user FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 索引
CREATE INDEX idx_exams_user_id ON exams(user_id);
CREATE INDEX idx_exams_exam_date ON exams(exam_date);
CREATE INDEX idx_exams_analysis_status ON exams(analysis_status);
```

**字段说明**：
- `subject_scores`：JSONB格式，灵活存储各科分数（不同考试科
- `analysis_status`：AI分析状态（pending=待分析，done=已完成，failed=失败）
- `standard_score`：折算750分制，方便跨考试比较进步情况

## 3.1 Claude API调用封装

**文件位置**：`F:\life-planner\backend\app\services\claude_service.py`

```python
import anthropic
from typing import AsyncGenerator, List, Dict
import os

class ClaudeService:
    def __init__(self):
        self.client = anthropic.AsyncAnthropic(
            api_key=os.getenv("ANTHROPIC_API_KEY")
        )
        self.model = "claude-sonnet-4-20250514"  # Claude Sonnet 4.6
    
    async def chat_stream(
        self, 
        messages: List[Dict], 
        system: str = ""
    ) -> AsyncGenerator[str, None]:
        """流式对话（用于AI教练页面）"""
        async with self.client.messages.stream(
            model=self.model,
            max_tokens=4096,
            system=system,
            messages=messages,
        ) as stream:
            async for event in stream:
                if event.type == "content_block_delta":
                    yield event.delta.text
    
    async def analyze_exam(
        self, 
        exam_data: Dict, 
        history_exams: List[Dict]
    ) -> Dict:
        """分析考试（非流式，返回完整JSON）"""
        prompt = self._build_exam_analysis_prompt(exam_data, history_exams)
        response = await self.client.messages.create(
            model=self.model,
            max_tokens=2048,
            messages=[{"role": "user", "content": prompt}],
        )
        return self._parse_json_response(response.content[0].text)
    
    def _build_exam_analysis_prompt(self, exam_data, history_exams) -> str:
        """构建考试分析Prompt"""
        return f"""
你是用户的AI学习教练。分析以下考试数据，输出JSON格式。

## 本次考试
{exam_data}

## 历史考试（用于对比进步情况）
{history_exams}

## 输出格式（严格JSON，不要多余文字）
{{
  "summary": "本次考试总结（50字）",
  "strengths": ["优势点1", "优势点2"],
  "weaknesses": ["薄弱点1", "薄弱点2"],
  "progress": "与上次考试对比（进步/退步/持平）",
  "suggestions": ["建议1", "建议2", "建议3"],
  "next_goal": "下次考试目标（具体可衡量）"
}}
"""

    def _parse_json_response(self, text: str) -> Dict:
        """解析AI返回的JSON（容错处理）"""
        import json, re
        # 去掉可能的markdown代码块标记
        text = re.sub(r'^```json\s*', '', text.strip(), flags=re.MULTILINE)
        text = re.sub(r'\s*```$', '', text, flags=re.MULTILINE)
        try:
            return json.loads(text)
        except json.JSONDecodeError:
            # 尝试提取JSON部分
            match = re.search(r'\{.*\}', text, re.DOTALL)
            if match:
                return json.loads(match.group())
            raise ValueError(f"无法解析AI返回的JSON: {text}")
```

## 3.2 Prompt模板设计（核心场景）

### 场景1：考试分析Prompt

```
# 角色
你是用户的高中学习教练，擅长从考试数据中发现问题并给出可执行的建议。

# 输入
- 本次考试：{exam_name}，总分{total_score}/{total_full}（标准分{standard_score}/750）
- 各科分数：{subject_scores_json}
- 历史考试：{history_exams_json}（用于对比进步）

# 任务
1. 分析本次考试的强项和弱项
2. 与历史考试对比，判断进步/退步
3. 给出3条具体可执行的改进建议
4. 设定下次考试目标（具体、可衡量）

# 输出格式（严格JSON，不要markdown）
{
  "summary": "...",
  "strengths": [...],
  "weaknesses": [...],
  "progress": "进步/退步/持平",
  "suggestions": [...],
  "next_goal": "..."
}
```

### 场景2：高考志愿推荐Prompt

```
# 角色
你是高考志愿填报专家，熟悉中国所有高校和专业的录取数据。

# 输入
- 用户分数：{user_score}分（标准分，满分750）
- 用户位次：{user_rank}名（{province}省{year}年）
- 用户选科：{user_subjects}（如"物理+化学+生物"）
- 用户偏好：{user_preferences}（如"想去一线城市"、"想学计算机"）

# 任务
1. 根据分数+位次，匹配近3年录取数据，推荐20个院校专业组合
2. 分成"冲刺"（录取概率30-50%）、"稳妥"（50-80%）、"保底"（80-95%）三档
3. 每个推荐附带"录取概率"和"推荐理由"
4. 标注数据来源（哪一年、哪个省、哪个学校）

# 数据约束（必须遵守）
- 只能使用数据库中的真实录取分数，禁止编造分数
- 录取概率 = (用户位次 - 院校录取位次) / 用户位次（简易公式）
- 如果数据不足，标注"数据不足，建议咨询学校招生办"

# 输出格式（严格JSON）
{
  "user_profile": {...},
  "recommendations": [
    {
      "college": "浙江大学",
      "major": "计算机科学与技术",
      "probability": "冲刺",  # 冲刺/稳妥/保底
      "score_match": 680,  # 该专业去年录取分（标准分）
      "reason": "你的分数接近去年录取线，有希望冲刺",
      "data_source": "2024年浙江省录取数据"
    }
  ],
  "analysis": "总体分析（100字）"
}
```

```

## 3.3 Token成本控制策略

| 策略 | 实现方式 | 效果 |
|------|---------|------|
| **Prompt压缩** | 只传必要字段，去掉冗长历史 | 省30-50% token |
| **上下文窗口管理** | 只保留最近5次对话，超出的摘要后传入 | 省50-70% token |
| **缓存Prompt** | Claude支持Prompt Caching（beta） | 省90%成本（重复Prompt部分） |
| **批量处理用廉价模型** | 数据清洗/批量分析用DeepSeek-V3 | 省90%成本 |
| **设置max_tokens上限** | 不同场景设不同上限（分析=2048，对话=4096） | 防止AI啰嗦 |

**成本估算（中性情况）**：
- 每用户每月活跃20天，每天对话5次
- 每次对话平均~1000 input token + ~500 output token
- Claude Sonnet 4.6价格：$3/1M input, $15/1M output
- 每用户每月成本 = 20×5×(1000×$3/1M + 500×$15/1M) ≈ $0.015/月
- 1000活跃用户/月 = $15/月 ≈ 110元/月（可接受）

## 4.3 Nginx配置（示例）

```nginx
# /etc/nginx/sites-available/life-planner
server {
    listen 443 ssl;
    server_name yourdomain.com;

    # SSL证书（阿里云免费证书）
    ssl_certificate /etc/nginx/ssl/yourdomain.com.pem;
    ssl_certificate_key /etc/nginx/ssl/yourdomain.com.key;

    # 前端静态文件
    location / {
        root /var/www/life-planner/frontend/dist;
        try_files $uri $uri/ /index.html;
    }

    # 后端API
    location /api/ {
        proxy_pass http://localhost:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # 媒体文件（用户上传的错题图片，OCR后删除，此目录应为空）
    location /media/ {
        alias /var/www/life-planner/media/;
    }
}

# HTTP跳转HTTPS
server {
    listen 80;
    server_name yourdomain.com;
    return 301 https://$server_name$request_uri;
}
```

## 4.4 部署流程（手动部署，MVP阶段）

```bash
# 1. 本地构建前端
cd F:\life-planner\frontend
npm run build
# 产出：dist/ 目录

# 2. 打包后端+前端
cd F:\life-planner
tar -czf deploy.tar.gz backend/ frontend/dist/

# 3. 上传到阿里云ECS
scp deploy.tar.gz root@your-ecs-ip:/tmp/

# 4. SSH登录ECS
ssh root@your-ecs-ip

# 5. 解压+安装依赖
cd /var/www
tar -xzf /tmp/deploy.tar.gz
cd backend
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 6. 启动后端（用systemd管理）
sudo systemctl start life-planner-api
sudo systemctl enable life-planner-api

# 7. 配置Nginx
sudo cp nginx.conf /etc/nginx/sites-available/life-planner
sudo ln -s /etc/nginx/sites-available/life-planner /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl restart nginx

# 8. 访问测试
curl https://yourdomain.com/api/health
```
